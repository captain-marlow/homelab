# Homelab — Ideas

*Loose pool of not-yet-projects. No order, no status discipline. When something firms up into a concrete next step, promote it to `projects.md` and delete it here.*

**Last updated:** 2026-06-24

---

- **Public / friends-and-family Matrix** — someday-goal, not a priority. Would mean flipping federation on (currently off; off-now/on-later is the clean direction).
- **Tailscale** — optional layer on top of the existing WireGuard remote access. Not a dependency for anything current.
- **AMD GPU for Ollama** — deferred due to VRAM limits and ROCm complexity. CPU/RAM tier first.
- **Architect on a local model** — likely a non-starter: the architect's value is frontier reasoning, which CPU-local models can't match, so the expected answer is *stays on Opus*. The local tier is better aimed at cheap, narrow work (heartbeat, classification, routing hints), not the planner. Revisit only with a GPU + a much larger open model.
- **Open WebUI for the Ollama tier** — browser chat front-end layered over Ollama (CT172) for ad-hoc human chat against local models, plus model management. Separate add-on; not needed for the embeddings scope (P005 deployed only the bare `ollama/ollama` server). Stand up once there's a local chat model worth talking to. https://github.com/open-webui/open-webui
- **Hermes ACP in VS Code** — wire Hermes into VS Code via its ACP integration, so the Mac-side executor is reachable from the editor. Pairs with the Hermes track (P006). https://hermes-agent.ai/integrations/vscode
- **Pin the Ollama LXC to performance (P) cores** — pve01's CPU is a hybrid P-core/E-core design, so a raw vCPU/thread count is a poor proxy for speed: llama.cpp inference threads that land on slower E-cores drag latency and make it inconsistent. Pin CT172 to P-core host CPUs via `lxc.cgroup2.cpuset.cpus` in `/etc/pve/lxc/172.conf` (identify which host CPU IDs are P-cores first, e.g. `lscpu -e`). Tune alongside the Ollama-tier core-count work; applies to any latency-sensitive local inference.
- **Second Obsidian vault sync** — personal-notes vault cloud sync method still TBD (separate from the homelab-repo-mirror vault).
- **OpenClaw "prose"** — explore further: https://docs.openclaw.ai/prose
- **Ansible auto-registers new LXCs in Mac `~/.ssh/config`** — every LXC created via the Mac Ansible flow should append a `Host` block (alias / `HostName` / `User` / `IdentitiesOnly yes` / `IdentityFile`) matching the existing uniform pattern, so new containers are reachable by name immediately. A `blockinfile` task keyed on the alias. Belongs with the Ansible/Hermes repo-integration work (P006).
- **Ansible LXC baseline-hardening role** — every new LXC should come up hardened by default: push the Mac's public key into the container (`authorized_keys` for the admin user) for immediate key-based access; **disable root SSH login** (`PermitRootLogin no`) and password auth (key-only); create a non-root sudo user; plus a general pass (firewall, unattended security upgrades). A reusable `baseline`/`hardening` role applied to every container. Pairs with the SSH-config auto-add above; same Ansible/Hermes track (P006).
- **Read-only web ingest for the `architect` agent** — give the planner a web-fetch HTTP GET and/or web-search tool. A planner that cannot research while planning is half-blind; “read-only” should mean “can ingest, cannot mutate,” not “cannot learn.” A GET-only fetch is a read, not a system change, so it preserves the planner → executor → human-gate separation. Constraints: GET-only; no POST, write, or auth-bearing methods; no exec; treat all fetched content as untrusted prompt-injection surface; scope the capability exclusively to the architect tool allowlist. Source: arose while researching agentskills.io, which architect could not fetch.
- **OpenAI Daybreak — securing the world** — explore further: https://openai.com/index/daybreak-securing-the-world/
- **Documentation MCP for agents (e.g. Context7)** — give the homelab agents (Hermes/OpenClaw) an MCP server that serves up-to-date, version-specific library/framework docs on demand, so code and configs are written against current APIs instead of stale training data. Context7 (Upstash) is one candidate: https://github.com/upstash/context7. Pairs with Hermes's MCP support (`hermes mcp add`) and the docs track (P007/P008).
- **Improve the `architect` agent (it's underperforming)** — the architect isn't planning as well as it should; treat as a focused improvement pass, not just one tweak. Known gap: **no web / documentation lookup** — a planner that can't research while planning is half-blind. Connect the threads: the **read-only web-ingest** bullet above (GET-only fetch / web-search on the architect's allowlist) and the **documentation-MCP** bullet above (a locally-hosted doc-MCP would give the architect *version-accurate* docs without opening general web access — the safer, more targeted half of "let it research"). Also explore a **stylist** capability for the docs work (P007/P008) — enforcing house style on proposed rewrites — which **may not need to be a separate agent**: a **skill or tool** that the architect (or Hermes) loads could be enough, and is cheaper to maintain. Open question to scope first: *what* is making the architect weak — model/prompt/SOUL, the locked tool allowlist, missing context, or response-style (it's verbose/over-caveated; see P007's architect response-style spec)? Pairs with: read-only web ingest, documentation MCP, and P007.
